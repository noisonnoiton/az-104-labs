    Lab 91 - AKS App Deploy (Kustomize)¶

    목표

    Lab 90에서 배포한 AKS 클러스터 위에 쇼핑몰 마이크로서비스(8개 앱 + 3개 인프라)를 Kustomize로 배포하고, 서비스 간 통신과 외부 접근을 검증합니다.

    핵심 체험:

    1. Kustomize overlay — base manifest + cluster-specific overlay 구조 이해
    2. Microservice 구조 — API Gateway, BFF, 도메인 서비스, 메시지 큐 패턴
    3. Kubernetes Service & Ingress — LoadBalancer / Ingress를 통한 외부 노출
    4. OpenTelemetry — 각 서비스에 OTel Java Agent가 자동 계측(traces, metrics, logs)

    Architecture Diagram¶

    Service Overview¶

    Application Services (namespace: awesome)¶

    서비스 역할 MySQL Redis RabbitMQ 서비스 호출
    bff Frontend UI — 클라이언트용 API 집계 — ✅ session — apigateway 경유 (5개 서비스)
    apigateway Spring Cloud Gateway — API 라우팅 — — — → account, cart, order, payment, product
    account 사용자 계정 CRUD ✅ accounts ✅ session ✅ publish (accountTopic) —
    cart 장바구니 ✅ carts ✅ session ✅ publish (cartTopic) —
    order 주문 처리 ✅ order ✅ session ✅ publish (orderTopic) —
    payment 결제 처리 ✅ payments ✅ session ✅ publish (paymentTopic) —
    product 상품 카탈로그 ✅ products ✅ session ✅ publish (productTopic) —
    rabbitmq-consumer Saga orchestrator — 비동기 이벤트 처리 — — ✅ consume (all topics) apigateway 경유 (saga callback)

    Infrastructure (namespace: awesome-infra)¶

    서비스 타입 설명
    MySQL StatefulSet 주 데이터베이스 (account, cart, order, payment, product 각각 별도 DB)
    RabbitMQ Deployment 이벤트 기반 Saga — 도메인 이벤트 pub/sub
    Redis Deployment HTTP Session 저장소 (Spring Session, 모든 서비스 공용)

    배포 구조 — Kustomize¶

    gitops-manifests/
├── base/                          # 공통 템플릿
│   ├── awesome/                   # Deployment + Service 템플릿
│   └── awesome-infra/             # MySQL, RabbitMQ, Redis
└── clusters/
    └── az104-aks/      # 클러스터별 overlay
        ├── awesome/               # 8개 서비스 (image, env, config patch)
        │   ├── account/
        │   ├── apigateway/        # + ingress.yaml
        │   ├── bff/
        │   ├── cart/
        │   ├── order/
        │   ├── payment/
        │   ├── product/
        │   └── rabbitmq-consumer/
        └── awesome-infra/         # MySQL password patch 등

    핵심 개념:

    • base/awesome/deployment.yaml — 공통 Deployment 스켈레톤 (image: awesome, port: 8080)
    • 각 overlay에서 nameSuffix, images, patches, configMapGenerator로 서비스별 설정 주입
    • OTel Java Agent가 각 서비스에 환경변수로 설정되어 자동 계측

    Prerequisite¶

    항목 설명
    AKS 클러스터 Lab 90 완료, kubectl get nodes 정상
    Ingress Controller Web App Routing 활성화 확인 — kubectl get ingressclass 에서 webapprouting.kubernetes.azure.com 존재
    ACR 공유 ACR에 서비스 이미지가 push 되어 있어야 함 (강사 사전 준비)
    kubectl AKS 자격 증명 설정 완료 (az aks get-credentials)
    매니페스트 gitops-manifests 저장소 clone

    매니페스트 저장소 Clone¶

    git clone https://github.com/noisonnoiton-shop/gitops-manifests.git
cd gitops-manifests

    Step-by-Step¶

    1) AKS 자격 증명 확인¶

    az aks get-credentials -g <RG_NAME> -n <AKS_NAME> --overwrite-existing
kubectl get nodes

    2) 매니페스트 준비 — Ingress IP 확인 및 nip.io host 설정¶

    nip.io란?

    nip.io는 무료 wildcard DNS 서비스입니다.
    <anything>.<IP>.nip.io로 요청하면 자동으로 <IP>로 resolve됩니다.

    호스트 해석되는 IP
    apigateway.awesome.1.2.3.4.nip.io 1.2.3.4
    awesome.1.2.3.4.nip.io 1.2.3.4

    별도 DNS 레코드 등록 없이 Ingress host-based routing을 즉시 사용할 수 있어 실습 환경에 적합합니다.

    Ingress manifest에 INGRESS_IP placeholder가 들어 있으므로, apply 전에 실제 IP로 치환해야 합니다.

    Ingress Controller External IP 확인¶

    # Web App Routing이 프로비저닝한 Service의 External IP 확인
kubectl get svc -n app-routing-system
    예상 출력 
    NAME    TYPE           CLUSTER-IP   EXTERNAL-IP   PORT(S)
nginx   LoadBalancer   172.16.x.x   20.39.x.x     80:3xxxx/TCP,443:3xxxx/TCP

    Ingress YAML의 placeholder 치환¶

    # 방법 A: 자동 감지
INGRESS_IP=$(kubectl get svc -n app-routing-system nginx \
  -o jsonpath='{.status.loadBalancer.ingress[0].ip}')

# 방법 B: 수동 지정 (Cloud Shell 등에서 직접 입력)
# INGRESS_IP="20.39.x.x"

echo "Ingress IP: ${INGRESS_IP}"

# ingress.yaml 파일의 INGRESS_IP placeholder를 실제 IP로 치환
cd clusters/az104-aks
sed -i "s/INGRESS_IP/${INGRESS_IP}/g" awesome/apigateway/ingress.yaml
sed -i "s/INGRESS_IP/${INGRESS_IP}/g" awesome/bff/ingress.yaml

# 치환 결과 확인
grep "host:" awesome/apigateway/ingress.yaml awesome/bff/ingress.yaml

    # Ingress IP 확인
$INGRESS_IP = kubectl get svc -n app-routing-system nginx -o jsonpath='{.status.loadBalancer.ingress[0].ip}'

# 또는 수동 지정
# $INGRESS_IP = "20.39.x.x"

Write-Host "Ingress IP: $INGRESS_IP"

cd clusters/az104-aks
(Get-Content awesome/apigateway/ingress.yaml) -replace 'INGRESS_IP', $INGRESS_IP | Set-Content awesome/apigateway/ingress.yaml
(Get-Content awesome/bff/ingress.yaml) -replace 'INGRESS_IP', $INGRESS_IP | Set-Content awesome/bff/ingress.yaml

Select-String "host:" awesome/apigateway/ingress.yaml, awesome/bff/ingress.yaml
    치환 결과 
    awesome/apigateway/ingress.yaml:  - host: apigateway.awesome.20.39.x.x.nip.io
awesome/bff/ingress.yaml:  - host: awesome.20.39.x.x.nip.io

    macOS sed

    macOS의 sed -i는 백업 확장자가 필요합니다. 에러가 나면 sed -i '' "s/..." 형식을 사용하세요.

    3) 인프라 배포 (MySQL, RabbitMQ, Redis)¶

    kubectl apply -k ./awesome-infra/

    # 배포 확인
kubectl get all -n awesome-infra
    예상 출력 
    NAME                            READY   STATUS    RESTARTS   AGE
pod/mysql-0                     1/1     Running   0          60s
pod/rabbitmq-xxxxx              1/1     Running   0          60s
pod/redis-xxxxx                 1/1     Running   0          60s

NAME               TYPE        CLUSTER-IP     PORT(S)
service/mysql      ClusterIP   10.0.x.x       3306/TCP
service/rabbitmq   ClusterIP   10.0.x.x       5672/TCP,15672/TCP
service/redis      ClusterIP   10.0.x.x       6379/TCP

    MySQL 초기화 대기

    MySQL StatefulSet은 초기화에 30~60초 소요됩니다. Running + READY 1/1 상태를 확인한 후 앱을 배포하세요.

    kubectl apply -k ./awesome/

    # 배포 확인
kubectl get all -n awesome
    예상 출력 
    NAME                                         READY   STATUS    RESTARTS   AGE
pod/awesome-account-xxxxx                    1/1     Running   0          45s
pod/awesome-apigateway-xxxxx                 1/1     Running   0          45s
pod/awesome-bff-xxxxx                        1/1     Running   0          45s
pod/awesome-cart-xxxxx                       1/1     Running   0          45s
pod/awesome-order-xxxxx                      1/1     Running   0          45s
pod/awesome-payment-xxxxx                    1/1     Running   0          45s
pod/awesome-product-xxxxx                    1/1     Running   0          45s
pod/awesome-rabbitmq-consumer-xxxxx          1/1     Running   0          45s

    5) Pod 상태 확인¶

    # 모든 Pod이 Running인지 확인
kubectl get pods -n awesome -w

# 특정 Pod 로그 확인 (문제 발생 시)
kubectl logs -n awesome deployment/awesome-apigateway --tail=50

    6) Ingress 확인¶

    kubectl get ingress -n awesome
    예상 출력 
    NAME                  CLASS                                HOSTS                                      ADDRESS        PORTS
awesome-apigateway    webapprouting.kubernetes.azure.com   apigateway.awesome.20.39.x.x.nip.io        20.39.x.x     80
awesome-bff           webapprouting.kubernetes.azure.com   awesome.20.39.x.x.nip.io                   20.39.x.x     80

    Web App Routing

    이 Lab에서는 AKS의 Web App Routing add-on (webapprouting.kubernetes.azure.com)을 사용합니다. Ingress Controller가 자동으로 Azure Load Balancer를 프로비저닝합니다.

    7) 서비스 접근 테스트¶

    # apigateway health check
curl -s "http://apigateway.awesome.${INGRESS_IP}.nip.io/actuator/health"

    curl -s "http://apigateway.awesome.%INGRESS_IP%.nip.io/actuator/health"
    예상 결과 
    {"status":"UP"}

    8) 서비스 간 통신 확인¶

    # apigateway를 통해 product 서비스 호출
curl -s "http://apigateway.awesome.${INGRESS_IP}.nip.io/api/products" | head -c 200

    curl -s "http://apigateway.awesome.%INGRESS_IP%.nip.io/api/products"

    # account 서비스 health (Pod 내부 통신 확인)
kubectl exec -n awesome deployment/awesome-apigateway -- curl -s http://awesome-account:8080/actuator/health

    Verify - Azure Portal¶

    1. AKS 클러스터 → Workloads → namespace awesome, awesome-infra에서 Deployment/StatefulSet 상태 확인
    2. AKS 클러스터 → Services and ingresses → Ingress awesome-apigateway 확인, External IP 확인
    3. AKS 클러스터 → Workloads → 각 Pod의 Logs 탭에서 Spring Boot 시작 로그 확인

    추가 확인 (Portal)

    • AKS → Workloads → awesome-infra namespace의 mysql StatefulSet이 1/1 Ready인지 확인
    • AKS → Services and ingresses → 서비스 목록에서 모든 ClusterIP 서비스가 Endpoint를 가지는지 확인
    • Load Balancer (MC_ 리소스 그룹) → Ingress에 의해 생성된 LB Frontend IP 확인

    OTel 계측 확인¶

    각 서비스에는 OpenTelemetry Java Agent가 -javaagent 옵션으로 설정되어 있습니다.

    kubectl get deployment -n awesome awesome-apigateway \
  -o jsonpath='{.spec.template.spec.containers[0].env}' | python3 -m json.tool | grep -A1 OTEL

    kubectl get deployment -n awesome awesome-apigateway -o jsonpath="{.spec.template.spec.containers[0].env}" | python -m json.tool | findstr OTEL
    환경변수 값 설명
    OTEL_EXPORTER_OTLP_ENDPOINT http://otel-apps.otel-collector:4318 OTel Collector 주소
    OTEL_SERVICE_NAME awesome-{service} 서비스 식별자
    OTEL_INSTRUMENTATION_MICROMETER_ENABLED true Micrometer 메트릭 수집
    OTEL_INSTRUMENTATION_LOGBACK_ENABLED true 로그 자동 수집

    OTel Collector 필요

    앱이 정상 동작하려면 otel-collector namespace에 OTel Collector가 배포되어 있어야 합니다. 배포되지 않아도 앱은 동작하지만, 텔레메트리 전송 시 connection refused 에러 로그가 발생합니다.
    → Lab 92 - AKS Monitoring에서 OTel Collector + Azure Monitor 연동을 설정합니다.

    Troubleshooting¶

    증상 원인 해결
    Pod ImagePullBackOff ACR 이미지 없음 또는 pull 권한 부족 az acr repository list, AKS-ACR role assignment 확인
    Pod CrashLoopBackOff 설정 오류 (DB 연결 등) kubectl logs 확인, ConfigMap 내용 점검
    MySQL Pod Pending PVC 프로비저닝 실패 StorageClass 확인, kubectl describe pvc
    Ingress IP 미할당 Web App Routing add-on 미활성화 az aks show로 add-on 상태 확인
    서비스 간 통신 실패 네임스페이스 간 DNS 미설정 FQDN 사용: mysql.awesome-infra.svc.cluster.local
    OTel 전송 에러 (connection refused) OTel Collector 미배포 Lab 92에서 배포 예정, 앱 동작에는 영향 없음
