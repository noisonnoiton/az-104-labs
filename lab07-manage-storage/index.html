    Lab 07 - Manage Azure Storage (Terraform)¶

    목표

    공식 Lab 07 - Manage Azure Storage에서 다루는 Storage Account, Blob, Azure Files, 네트워크 액세스 개념을 Terraform으로 배포하고, Portal / CLI로 검증합니다.

    핵심 체험:

    1. Blob Public Access — index.html을 브라우저에서 공개 접근
    2. Network Access 제한 — Deny 전환 후 브라우저에서 403, VNet 내 VM에서는 접근 성공 (Service Endpoint)
    3. Blob vs File Share — Storage Explorer에서 유사하게 보이지만 내부 저장 방식이 다른 두 서비스 비교

    Architecture Diagram¶

    Resources Created¶

    리소스 기본 이름/값
    VNet vnet-{prefix} (10.30.0.0/16)
    Subnet snet-vm (10.30.0.0/24), Service Endpoint: Microsoft.Storage
    NSG nsg-vm-{prefix} — SSH(22) 허용
    NSG Rule AllowSSH (priority 100)
    NSG-Subnet 연결 snet-vm ↔ NSG
    Public IP pip-vm-{prefix} (Standard, Static)
    NIC nic-vm-{prefix}
    VM vm-{prefix} (Ubuntu 24.04 LTS, Standard_D2s_v3)
    Storage Account st{prefix}{random} (전역 고유, LRS)
    Blob Container web (public read — anonymous blob access)
    Blob (HTML) web/index.html — 테스트용 정적 페이지
    File Share share01 (5 GiB quota)
    random_string Storage Account 이름 suffix (6자)
    random_password VM admin password (16자)

    비용 주의

    VM(Standard_D2s_v3)과 Storage Account가 과금됩니다. 실습 후 terraform destroy로 정리하세요.

    Prerequisite¶

    항목 설명
    권한 할당된 Resource Group에 Contributor 이상
    도구 Azure CLI 설치, Terraform 1.6+ 설치
    인증 az login 후 올바른 subscription 선택
    선행 Lab Lab 00 - Skeleton으로 RG 연결 검증 권장

    Concept Mapping (Official Lab 07)¶

    공식 Lab 07 주제 이 모듈에서의 위치
    Storage Account 프로비저닝 storage.tf — azurerm_storage_account (tier, replication)
    Blob 컨테이너 + Public Access storage.tf — web container (blob access) + index.html 업로드
    Azure Files storage.tf — azurerm_storage_share (SMB, quota)
    네트워크 액세스 제어 network.tf — VNet + Service Endpoint, storage.tf — network_rules (Allow/Deny)
    VNet 내부 접근 검증 vm.tf — VNet 내 VM에서 curl로 Blob 접근 (Service Endpoint 경유)
    SAS (Shared Access Signature) Terraform 범위 밖 — Portal/CLI로 직접 생성하여 검증

    Blob vs File Share — 차이점¶

    Portal의 Storage Explorer에서 보면 Blob Container와 File Share 모두 디렉터리/파일 구조로 탐색할 수 있어서 비슷해 보입니다. 하지만 내부 동작은 다릅니다.

    항목 Blob Storage Azure Files (File Share)
    프로토콜 HTTP/HTTPS (REST API) SMB / NFS / REST
    저장 구조 Flat namespace — "디렉터리"는 이름에 /가 포함된 blob (가상 경로) 실제 디렉터리 + 파일 (POSIX 또는 SMB 시맨틱)
    접근 방식 URL로 직접 접근 (public container 가능) 마운트 (mount -t cifs, Windows 드라이브 매핑)
    주 용도 정적 웹, 이미지, 로그, 백업, Data Lake VM 공유 폴더, 레거시 앱의 파일 시스템
    디렉터리 존재하지 않음 — blob 이름이 images/photo.jpg이면 Explorer에서 images/ 폴더처럼 표시될 뿐 실제 디렉터리 생성/삭제 가능
    Lock / 동시 접근 blob lease (단일 writer) SMB oplock, 다중 사용자 동시 접근

    Storage Explorer에서 확인

    Portal에서 Storage Explorer → web 컨테이너와 share01 File Share를 각각 열어 보세요. 둘 다 파일 목록이 보이지만, Blob에는 실제 디렉터리 개념이 없고 가상 경로(/로 구분된 이름)만 있습니다.

    Step-by-Step¶

    Terraform 코드 위치: labs/terraform/lab07-manage-storage/

    1) Prepare variables¶

    cd labs/terraform/lab07-manage-storage
cp terraform.tfvars.example terraform.tfvars

    에디터로 terraform.tfvars를 열고 다음을 설정합니다.

    • resource_group_name (필수) — 강사가 알려준 본인 RG 이름

    2) Login to Azure¶

    az login
az account show -o table

    3) Initialize Terraform¶

    terraform init

    4) Plan¶

    terraform plan

    terraform plan에서 볼 것

    • 추가될 리소스 수 — 총 14개 (VNet, Subnet, NSG, NSG Rule, NSG-Subnet, PIP, NIC, VM, Storage Account, Container, Blob, File Share, random_string, random_password)
      • Storage Account 이름 — (known after apply) 로 표시되며, random_string이 적용 후 확정됨
      • network_rules — default_action = "Allow", virtual_network_subnet_ids에 snet-vm 서브넷 ID 포함
      • web 컨테이너 — container_access_type = "blob" (anonymous read)

    5) Apply¶

    terraform apply

    프롬프트에 yes를 입력합니다.

    예상 출력 (요약) 
    Apply complete! Resources: 14 added, 0 changed, 0 destroyed.

Outputs:
file_share_name        = "share01"
index_html_url         = "https://staz104lab07xxxxxx.blob.core.windows.net/web/index.html"
location               = "koreacentral"
network_default_action = "Allow"
primary_access_key     = <sensitive>
primary_blob_endpoint  = "https://staz104lab07xxxxxx.blob.core.windows.net/"
resource_group_name    = "rg-az104-student01"
storage_account_name   = "staz104lab07xxxxxx"
vm_admin_password      = <sensitive>
vm_admin_username      = "azureuser"
vm_public_ip           = "x.x.x.x"
vnet_name              = "vnet-az104-lab07"
web_container_name     = "web"

    6) Check outputs¶

    terraform output
terraform output index_html_url
terraform output vm_public_ip

    Blob Public Access 테스트¶

    Terraform이 web 컨테이너에 index.html을 업로드했습니다. container_access_type = "blob"이므로 인증 없이 URL만으로 접근 가능합니다.

    브라우저에서 접근¶

    terraform output index_html_url

    출력된 URL을 브라우저에 붙여넣습니다.

    예상 결과

    브라우저에 "Hello from AZ-104 Lab 07!" 페이지가 표시됩니다.

    curl로 확인¶

    curl -s "$(terraform output -raw index_html_url)"

    for /f "tokens=*" %a in ('terraform output -raw index_html_url') do curl -s "%a"

    Network Access 제한 테스트¶

    1단계: Deny로 전환¶

    terraform.tfvars에 다음을 추가합니다.

    network_default_action = "Deny"

    terraform plan    # network_rules.default_action: "Allow" → "Deny" 변경 확인
terraform apply

    2단계: 공개 접근 차단 확인 (수강생 브라우저)¶

    같은 URL을 브라우저에서 다시 접속합니다.

    curl -s -o /dev/null -w "%{http_code}" "$(terraform output -raw index_html_url)"

    for /f "tokens=*" %a in ('terraform output -raw index_html_url') do curl -s -o NUL -w "%%{http_code}" "%a"
    예상 결과 

    403
    HTTP 403 Forbidden — 수강생 노트북의 공인 IP가 허용 목록에 없으므로 차단됩니다.

    3단계: VNet 내 VM에서 접근 확인 (Service Endpoint)¶

    VM에 SSH로 접속합니다. 로컬 터미널 또는 Azure Portal Cloud Shell을 사용합니다.

    방법 A — 로컬 터미널

    ssh azureuser@$(terraform output -raw vm_public_ip)
# 비밀번호: terraform output -raw vm_admin_password

    for /f "tokens=*" %a in ('terraform output -raw vm_public_ip') do ssh azureuser@%a
REM 비밀번호: terraform output -raw vm_admin_password

    방법 B — Azure Cloud Shell (Bash)

    로컬에서 SSH가 안 되는 경우 Azure Portal → 상단 Cloud Shell 아이콘 (>_) → Bash 를 선택합니다.

    # Cloud Shell에서 실행 (VM Public IP를 복사해서 입력)
ssh azureuser@<VM_PUBLIC_IP>

    Cloud Shell에서 비밀번호 입력

    Terraform output은 Cloud Shell에서 직접 실행할 수 없으므로, 로컬 터미널에서 terraform output -raw vm_admin_password와 terraform output -raw vm_public_ip로 값을 먼저 복사한 뒤, Cloud Shell에 붙여넣으세요.

    VM 안에서 같은 blob URL에 접근합니다.

    # VM 내부에서 실행
curl -s "https://<storage_account_name>.blob.core.windows.net/web/index.html"
    예상 결과 

    <!DOCTYPE html>
<html lang="en">
<head><meta charset="UTF-8"><title>AZ-104 Lab 07</title></head>
<body>
  <h1>Hello from AZ-104 Lab 07!</h1>
  ...
</body>
</html>
    HTTP 200 OK — VM은 snet-vm 서브넷의 Service Endpoint 경유로 Storage에 접근하므로, Deny 규칙에도 불구하고 정상 접근됩니다.

    핵심 포인트

    접근 위치 default_action = Allow default_action = Deny
    수강생 브라우저 (공인 IP) ✓ 200 ✗ 403
    VNet 내 VM (Service Endpoint) ✓ 200 ✓ 200

    같은 Storage Account, 같은 URL인데 접근 경로에 따라 결과가 다릅니다. 이것이 Service Endpoint의 핵심입니다.

    4단계: Allow로 복원¶

    terraform.tfvars에서 해당 줄을 삭제하거나 주석 처리합니다.

    # network_default_action = "Deny"

    terraform apply

    브라우저에서 다시 접근하면 index.html이 정상 표시됩니다.

    반영 지연

    Network rule 변경 후 실제 적용까지 최대 30초~1분 걸릴 수 있습니다. 즉시 되지 않으면 잠시 기다렸다 다시 시도하세요.

    Verify - Azure Portal¶

    1. Resource Group → Storage Account, VM, VNet 등이 보이는지 확인합니다.
    2. Storage Account → Containers → web (Blob access level: Blob) 확인.
    3. Storage Account → Containers → web → index.html이 업로드되어 있는지 확인.
    4. Storage Account → File shares → share01이 존재하는지, quota가 5 GiB인지 확인.
    5. Storage Account → Networking → "Firewalls and virtual networks"에서 snet-vm 서브넷이 등록되어 있는지, default action 값 확인.

    Storage Explorer로 Blob vs File Share 비교¶

    Portal에서 Storage Explorer (preview) 를 열고:

    • Blob Containers → web → index.html 파일 확인. 추가 파일 업로드, 가상 폴더 생성을 시도해 보세요.
    • File Shares → share01 → 새 디렉터리 생성, 파일 업로드를 시도해 보세요.

    양쪽 다 "폴더/파일"처럼 보이지만, Blob에서 만든 "폴더"는 실제로는 이름에 /가 포함된 blob이고, File Share에서 만든 폴더는 실제 디렉터리입니다.

    추가 확인

    Portal에서 더 자세히 확인하고 싶다면:

    • Storage Account → Access keys 에서 Key / Connection string 확인
    • Storage Account → Networking → Network Routing 탭에서 라우팅 설정 확인
    • Activity log 에서 최근 변경 이력 (network rule 변경 등) 확인

    Cleanup (Teardown)¶

    terraform destroy

    프롬프트에 yes를 입력합니다.

    데이터 삭제

    Storage Account가 삭제되면 그 안의 모든 blob, file share 데이터가 영구 삭제됩니다.

    Terraform Files¶

    파일 역할
    versions.tf Terraform / provider 버전 제약
    providers.tf azurerm provider
    variables.tf prefix, RG, replication, network action, VM size 등
    main.tf RG data source, random_string, locals
    network.tf VNet, Subnet (snet-vm + Microsoft.Storage Service Endpoint)
    vm.tf Ubuntu VM, NIC, Public IP, NSG (SSH only), random_password
    storage.tf Storage Account, web Blob Container, index.html Blob, File Share
    outputs.tf Storage name/URL, VM IP/password, network action
    terraform.tfvars.example 변수 예시 파일 (비밀 값 없음)

    Troubleshooting¶

    증상 원인 해결
    Storage Account 이름 충돌 전역 고유 이름이어야 함 random_string이 자동 생성하므로 재시도
    AuthorizationFailure blob 업로드 RBAC 권한 부족 또는 network rule 차단 --auth-mode login 사용, network rules 확인
    Service Endpoint 미동작 Subnet에 endpoint 미설정 az network vnet subnet show로 serviceEndpoints 확인
    403 on public blob (Allow 상태) allow_nested_items_to_be_public = false storage.tf에서 true로 변경 후 재apply
    Deny 전환 후에도 접근 가능 Network rule 반영 지연 (최대 1분) 30초~1분 후 재시도
    Allow 복원 후에도 차단 브라우저 캐시 또는 반영 지연 시크릿 탭으로 시도, 1분 대기
    VM SSH 접속 불가 NSG rule 누락 또는 PIP 미할당 az network nsg rule list, az vm list-ip-addresses 확인

    Cost & Caution¶

    실수 방지

    terraform.tfvars의 resource_group_name이 반드시 본인 RG인지 확인한 뒤 apply하세요. Storage Account 삭제 시 데이터는 복구할 수 없습니다.
