    Lab 92 - AKS Monitoring (Terraform + OTel)¶

    목표

    AKS에 배포된 마이크로서비스의 Traces, Metrics, Logs를 OpenTelemetry Collector를 통해 Azure Monitor (Application Insights)로 전송하고, Portal에서 분산 추적과 메트릭을 확인합니다.

    핵심 체험:

    1. Azure Monitor 백엔드 생성 — Log Analytics Workspace + Application Insights (Terraform)
    2. OTel Collector 배포 — Kubernetes에 Collector 배포, Azure Monitor exporter 연동
    3. 분산 추적 (Distributed Tracing) — Application Insights Transaction Search에서 end-to-end trace 확인
    4. 메트릭 & 로그 — Live Metrics, Log Analytics 쿼리 (KQL)
    5. Alert 생성 — 에러율 기반 Alert Rule 설정

    Architecture Diagram¶

    Resources Created (Terraform)¶

    리소스 이름 설명
    Log Analytics Workspace law-{prefix} 로그, 메트릭 저장소 (KQL 쿼리 대상)
    Application Insights appi-{prefix} 분산 추적, Live Metrics, 성능 모니터링

    비용

    Log Analytics는 수집 데이터량 기반 과금 (PerGB2018). 실습 규모에서는 거의 무료에 가깝습니다 (Free tier: 5GB/월).

    Step-by-Step¶

    Part 1: Azure Monitor 백엔드 생성 (Terraform)¶

    Terraform 환경에서 실행

    Part 1은 Terraform이 구성된 환경 (로컬 터미널 또는 lab 환경)에서 실행합니다.
    Cloud Shell에는 Terraform state가 없으므로 terraform 명령이 동작하지 않습니다.

    0) Container Insights 활성화 (AKS)¶

    AKS 클러스터에 Container Insights를 활성화하면 노드/Pod CPU, Memory, 컨테이너 로그를 Log Analytics로 자동 수집합니다.

    AKS_NAME=$(cd ../lab90-aks && terraform output -raw aks_cluster_name)
RG=$(cd ../lab90-aks && terraform output -raw resource_group_name)

az aks enable-addons --addons monitoring \
  -g "$RG" -n "$AKS_NAME" \
  --workspace-resource-id "$(terraform output -raw log_analytics_workspace_id)" \
  --no-wait

    # 본인 RG 이름과 AKS 이름을 직접 지정
RG="<본인 RG 이름>"
AKS_NAME="<AKS 클러스터 이름>"  # 예: aks-az104-lab90

# LAW 리소스 ID 조회
LAW_ID=$(az monitor log-analytics workspace show \
  -g "$RG" -n "law-az104-lab92" \
  --query id -o tsv)

az aks enable-addons --addons monitoring \
  -g "$RG" -n "$AKS_NAME" \
  --workspace-resource-id "$LAW_ID" \
  --no-wait

    Container Insights vs OTel

    Container Insights는 노드/Pod 수준 인프라 메트릭을 수집합니다 (CPU, Memory, Network, Disk).
    OTel Collector는 애플리케이션 수준 텔레메트리 (Traces, 앱 Metrics, Logs)를 수집합니다.
    둘 다 활성화하면 인프라 + 앱 양쪽을 한 곳에서 확인할 수 있습니다.

    1) Prepare variables¶

    cd labs/terraform/lab92-aks-monitoring
cp terraform.tfvars.example terraform.tfvars

    terraform.tfvars에서 resource_group_name을 본인 RG로 설정합니다.

    2) Deploy¶

    terraform init
terraform plan
terraform apply
    예상 출력 
    Apply complete! Resources: 2 added, 0 changed, 0 destroyed.

Outputs:
application_insights_connection_string = <sensitive>
application_insights_name              = "appi-az104-lab92"
log_analytics_workspace_name           = "law-az104-lab92"
...

    3) Connection String 확인¶

    terraform output -raw application_insights_connection_string

    Application Insights (appi-az104-lab92) → Overview → Connection String 복사

    az monitor app-insights component show \
  -g "<본인 RG 이름>" -a "appi-az104-lab92" \
  --query connectionString -o tsv

    이 값을 다음 단계에서 OTel Collector Secret에 주입합니다.

    앱 서비스들이 이미 otel-apps.otel-collector:4318로 데이터를 보내고 있으므로, Collector가 배포되면 즉시 데이터가 Azure Monitor로 흐릅니다.

    1) 트래픽 생성¶

    INGRESS_IP=$(kubectl get svc -n app-routing-system nginx \
  -o jsonpath='{.status.loadBalancer.ingress[0].ip}')

for i in {1..20}; do
  curl -s "http://apigateway.awesome.${INGRESS_IP}.nip.io/api/products" > /dev/null
  curl -s "http://apigateway.awesome.${INGRESS_IP}.nip.io/api/accounts" > /dev/null
done

echo "요청 20회 완료 — 2~3분 후 Application Insights에서 확인 가능"

    for /f "tokens=*" %a in ('kubectl get svc -n app-routing-system nginx -o jsonpath^="{.status.loadBalancer.ingress[0].ip}"') do set INGRESS_IP=%a

for /L %i in (1,1,20) do (
  curl -s "http://apigateway.awesome.%INGRESS_IP%.nip.io/api/products" > NUL
  curl -s "http://apigateway.awesome.%INGRESS_IP%.nip.io/api/accounts" > NUL
)

echo 요청 20회 완료

    2) Application Map — 서비스 의존 관계¶

    1. Application Insights → Investigate → Application map
    2. 서비스 간 의존 관계 그래프가 자동 생성됩니다 (apigateway → product, account, cart 등).
    3. 각 노드(서비스)를 클릭하면:
      • Investigate failures — 해당 서비스의 실패 분석
      • Investigate performance — 응답 시간 분석
      • Go to details — End-to-end transaction details (Gantt chart)

    Application Map

    Application Map은 분산 앱의 토폴로지를 시각화합니다. 각 노드에 평균 응답 시간, 실패율이 표시되어 병목이나 장애 서비스를 한눈에 파악할 수 있습니다.
    → Application map: Triage distributed applications

    3) Performance — 응답 시간 분석¶

    1. Application Insights → Investigate → Performance
    2. Operations 탭에서 전체 요청 목록, 평균 응답 시간, 호출 횟수를 확인합니다.
    3. 느린 요청을 클릭 → Drill into → 샘플 선택 → End-to-end transaction details 에서 호출 체인(Gantt chart) 확인

    Dependency 확인

    Performance 화면 하단의 Dependencies 탭에서 외부 호출(MySQL, Redis, RabbitMQ 등) 응답 시간도 확인할 수 있습니다.

    4) Failures — 실패 분석¶

    1. Application Insights → Investigate → Failures
    2. Operations 탭 — 실패한 요청의 HTTP 상태 코드별 분류
    3. Exceptions 탭 — 서비스에서 발생한 예외(Exception) 목록
    4. 항목 클릭 → End-to-end transaction details 에서 실패 원인 추적

    의도적 에러 생성

    실패 데이터를 확인하려면 존재하지 않는 endpoint로 요청을 보내세요:

    for i in {1..10}; do
  curl -s "http://apigateway.awesome.${INGRESS_IP}.nip.io/api/nonexistent"
done

    for /L %i in (1,1,10) do curl -s "http://apigateway.awesome.%INGRESS_IP%.nip.io/api/nonexistent"

    참고: Investigate failures, performance, and transactions

    5) Search — 개별 텔레메트리 검색¶

    1. Application Insights → Investigate → Search
    2. 시간 범위를 Last 30 minutes로 설정
    3. Event types 필터로 Request, Dependency, Exception, Trace 등 선택
    4. 항목 클릭 → End-to-end transaction details 에서 전체 호출 체인 확인

    Transaction Diagnostics

    End-to-end transaction details 화면은 Gantt chart 형태로 요청의 전체 흐름을 보여줍니다. 서버 간 텔레메트리를 자동으로 correlation하여 하나의 요청이 여러 서비스를 거치는 과정을 추적할 수 있습니다.

    6) Live Metrics (참고)¶

    1. Application Insights → Investigate → Live Metrics

    Live Metrics 미지원 (OTel Collector 경유 시)

    Live Metrics(QuickPulse)는 애플리케이션 SDK가 Application Insights에 직접 연결할 때만 동작합니다.
    이 Lab에서는 앱 → OTel Collector → Azure Monitor Exporter 경로를 사용하므로 Live Metrics가 표시되지 않습니다.
    이는 정상이며, 텔레메트리 데이터는 위의 Performance, Failures, Application Map, Search 등에서 확인할 수 있습니다.
    → Live Metrics 지원 조건

    Part 4: Logs (KQL) 쿼리¶

    Application Insights → Monitoring → Logs 에서 아래 KQL 쿼리를 실행합니다.

    데이터 지연

    Logs 쿼리 결과는 2~5분 지연이 있을 수 있습니다. 트래픽 생성 후 잠시 대기한 뒤 쿼리를 실행하세요.

    서비스별 요청 수¶

    requests
| where timestamp > ago(30m)
| summarize count() by cloud_RoleName
| order by count_ desc

    서비스별 평균 응답 시간 (P95 포함)¶

    requests
| where timestamp > ago(30m)
| summarize
    avg_duration = avg(duration),
    p95_duration = percentile(duration, 95),
    request_count = count()
  by cloud_RoleName
| order by avg_duration desc

    실패 요청 조회¶

    requests
| where timestamp > ago(1h)
| where success == false
| project timestamp, cloud_RoleName, name, resultCode, duration
| order by timestamp desc

    Dependency 호출 분석 (MySQL, Redis 등)¶

    dependencies
| where timestamp > ago(30m)
| summarize
    avg_duration = avg(duration),
    failure_count = countif(success == false),
    total_count = count()
  by target, type
| order by avg_duration desc

    분산 추적 — 특정 요청의 호출 체인¶

    requests
| where timestamp > ago(30m)
| where name contains "products"
| take 1
| project operation_Id

    위 쿼리 결과의 operation_Id를 복사한 뒤:

    union requests, dependencies, traces
| where operation_Id == "<복사한 operation_Id>"
| order by timestamp asc
| project timestamp, itemType, name, duration, success

    참고: Application Insights telemetry data model

    Part 5: Alert Rule 생성 (선택)¶

    1. Application Insights → Monitoring → Alerts → + Create → Alert rule
    2. Condition: Signal = Failed requests, Threshold > 5 (5분 내 실패 5건 이상)
    3. Actions: 이메일 알림 또는 Action Group (실습에서는 생략 가능)
    4. Details: Alert rule name = High Error Rate - Lab92
    5. Review + create

    OTel Collector 설정 설명¶

    otel-collector.yaml에 정의된 Collector 구성:

    구성 요소 설정 설명
    Receiver otlp (gRPC:4317, HTTP:4318) 앱에서 보내는 OTLP 데이터 수신
    Processor memory_limiter, batch 메모리 보호 + 배치 전송 (효율)
    Exporter azuremonitor Application Insights로 전송
    Exporter debug stdout 로그 (테스트용)

    운영 환경 권장 사항

    • debug exporter 제거
    • memory_limiter 값을 노드 리소스에 맞게 조정
    • Collector를 DaemonSet 또는 Deployment(replica 2+)로 운영
    • Sampling 설정 추가 (예: probabilistic_sampler, rate 0.1)

    참고: OTel Collector 없이 Azure Monitor로 직접 전송¶

    이 Lab에서는 OTel Collector를 별도 배포하여 Traces/Metrics/Logs를 Azure Monitor로 전송합니다. 하지만 Collector를 거치지 않는 직접 전송 방법도 있습니다.

    방법 1 — Application SDK 직접 Exporter¶

    각 언어별 Azure Monitor OpenTelemetry Exporter/Distro를 앱 코드에 추가하면, Application Insights의 Connection String만으로 Collector 없이 직접 전송할 수 있습니다.

    언어 패키지
    .NET Azure.Monitor.OpenTelemetry.AspNetCore → UseAzureMonitor()
    Java Azure Monitor OpenTelemetry Distro (auto-attach)
    Node.js @azure/monitor-opentelemetry
    Python azure-monitor-opentelemetry-exporter

    앱에 APPLICATIONINSIGHTS_CONNECTION_STRING 환경변수를 주입하면 traces, metrics, logs가 Application Insights로 바로 전송됩니다.

    참고: Enable Azure Monitor OpenTelemetry

    방법 2 — AKS Autoinstrumentation (Preview)¶

    AKS 클러스터 설정에서 Application Monitoring을 활성화하면, Java / Node.js 앱에 대해 코드 수정 없이 자동 계측이 가능합니다 (Preview).

    1. AKS Monitor settings → "Enable support for Autoinstrumentation" 활성화
    2. Application Insights 리소스에 "Enable OTLP Support (Preview)" 켬
    3. Namespace별로 autoinstrumentation 또는 autoconfiguration 선택

    별도 Collector 배포 없이, AKS 플랫폼 수준에서 OTel 데이터를 Azure Monitor로 라우팅합니다.

    참고: Monitor AKS applications with OTLP (Preview)

    Collector 방식과의 비교¶

    항목 OTel Collector (이 Lab) SDK 직접 Exporter AKS Autoinstrumentation
    추가 인프라 Collector Pod 필요 없음 없음 (AKS add-on)
    코드 변경 앱은 OTLP로 export → Collector가 변환 앱에 SDK/패키지 추가 필요 Java/Node.js는 변경 없음
    지원 언어 모든 OTel 호환 언어 .NET, Java, Node.js, Python Java, Node.js (Preview)
    유연성 높음 (필터, 샘플링, 멀티 백엔드) 중간 낮음 (플랫폼 관리)
    상태 GA (contrib exporter) GA Preview

    이 Lab에서 Collector를 사용하는 이유

    Collector 방식은 언어에 상관없이 동작하고, 필터링·샘플링·멀티 백엔드 전송 등 유연한 파이프라인 설계가 가능합니다. 운영 환경에서 가장 보편적인 패턴이며, 직접 Exporter나 AKS Autoinstrumentation은 단순한 구성이 필요할 때 적합합니다.

    Troubleshooting¶

    증상 원인 해결
    Collector CrashLoop Connection String 누락 Secret 생성 확인: kubectl get secret -n otel-collector
    Application Insights 데이터 없음 Exporter 설정 오류 Collector 로그 확인: kubectl logs -n otel-collector deployment/otel-apps
    Traces만 보이고 Metrics 없음 Pipeline 설정 누락 config.yaml에서 metrics pipeline 확인
    앱 로그에 "connection refused" Collector 미배포 또는 Service DNS 미해결 kubectl get svc -n otel-collector, endpoint 확인
    Log Analytics 쿼리 결과 없음 데이터 지연 (최대 5분) 5분 대기 후 재시도
    Alert 미발동 Threshold 미달 또는 평가 기간 미경과 Alert condition의 evaluation period, threshold 확인

    Cost & Caution¶

    데이터 수집 비용

    Log Analytics는 수집량 기반 과금입니다. 실습 후 OTel Collector를 삭제하지 않으면 계속 데이터가 쌓입니다. 실습 종료 시 반드시 terraform destroy와 Collector 정리를 수행하세요.
