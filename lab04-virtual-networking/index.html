    Lab 04 - Virtual Networking (Terraform)¶

    목표

    공식 Lab 04 - Implement Virtual Networking에서 다루는 VNet, Subnet, NSG, ASG, Peering, Private DNS, Bastion 개념을 Terraform으로 한 번에 배포하고, Portal / CLI로 검증합니다. 공식 Lab은 Windows Server VM 위주이지만, 이 구성은 Ubuntu 24.04로 동일한 네트워크 패턴을 연습합니다.

    Architecture Diagram¶

    Resources Created¶

    이 모듈에서 생성되는 Azure 리소스입니다.

    리소스 기본 이름/값
    Hub VNet / Spoke VNet vnet-{prefix}-hub / vnet-{prefix}-spoke (10.10.0.0/16, 10.20.0.0/16)
    Workload subnet Hub, Spoke 각 snet-workload (/24), 동일 NSG 연결
    AzureBastionSubnet 10.10.100.0/26 (Bastion 전용 이름 고정)
    Peering peer-hub-to-spoke, peer-spoke-to-hub (양방향)
    NSG nsg-{prefix}-workload — SSH from Bastion subnet, ICMP between VNets, HTTP to ASG
    ASG asg-{prefix}-app — 두 VM NIC에 연결
    Private DNS zone lab04.internal(변경 가능), VNet link x2, vm-hub / vm-spoke A record
    Bastion Basic SKU + Standard Public IP
    VM vm-hub, vm-spoke — Ubuntu 24.04, password auth (random_password)

    비용 주의

    Azure Bastion (Basic)과 VM 2대는 시간당 과금됩니다. 실습이 끝나면 반드시 terraform destroy로 정리하세요.

    Application Security Group (ASG) 이란?¶

    NSG 규칙에서 IP 주소 대신 논리적 그룹으로 대상을 지정할 수 있게 해주는 리소스입니다.

    비교 항목 IP 기반 규칙 ASG 기반 규칙
    규칙 작성 destination_address_prefix = "10.20.1.4" destination_application_security_group_ids = [asg.id]
    VM 추가 시 규칙에 IP를 추가 수정 필요 NIC를 ASG에 연결만 하면 됨
    VM 10대 확장 규칙 10개 수정 또는 IP 목록 관리 NIC 10개를 같은 ASG에 연결, 규칙 변경 없음
    유지보수 IP 변경 시 규칙도 변경 VM이 교체되어도 ASG 멤버십만 유지

    이 Lab에서는 HTTP 규칙(우선순위 120)에서 destination_application_security_group_ids를 사용합니다:

    • nsg-hub: asg-app-hub 멤버(vm-hub)의 포트 80만 허용
    • nsg-spoke: asg-app-spoke 멤버(vm-spoke)의 포트 80만 허용

    즉, 같은 서브넷에 다른 VM이 있더라도 ASG에 속하지 않으면 HTTP 트래픽을 받지 않습니다.

    VNet Peering 옵션 이해¶

    Azure VNet Peering에는 4가지 옵션이 있습니다. Portal에서는 한 화면에 양쪽(Local / Remote) 설정을 동시에 구성하지만, Terraform에서는 양방향 각각을 별도 리소스로 선언합니다.

    # network.tf — hub → spoke (이 Lab)
resource "azurerm_virtual_network_peering" "hub_to_spoke" {
  allow_virtual_network_access = true
  allow_forwarded_traffic      = false
  allow_gateway_transit        = false
  use_remote_gateways          = false
}

# network.tf — spoke → hub (이 Lab)
resource "azurerm_virtual_network_peering" "spoke_to_hub" {
  allow_virtual_network_access = true
  allow_forwarded_traffic      = false
  allow_gateway_transit        = false
  use_remote_gateways          = false
}

    1) allow_virtual_network_access — 피어링의 기본 목적¶

    피어링된 양쪽 VNet의 VM끼리 private IP로 직접 통신을 허용합니다. true가 아니면 피어링을 만든 의미가 없으므로 거의 항상 true 입니다.

    2) allow_forwarded_traffic — NVA 경유 트래픽 허용¶

    이 옵션이 가장 헷갈립니다. "제3자가 forward한 패킷이 Peering을 통과해도 되는가?" 를 결정합니다.

    핵심 전제: VNet Peering은 비전이적(Non-transitive)

    Spoke-1 ↔ Hub, Spoke-2 ↔ Hub가 각각 피어링되어 있어도, Spoke-1 → Spoke-2 경로는 자동으로 생기지 않습니다. Spoke-1의 패킷을 Hub의 NVA(방화벽)가 받아서 Spoke-2로 전달(forward) 해야 합니다.

    Spoke-1 → Hub(NVA) → Spoke-2 통신 흐름:

    Spoke-1 (10.1.0.0/16)        Hub VNet                     Spoke-2 (10.2.0.0/16)
    │                            │                              │
    │  src: 10.1.0.4             │                              │
    │  dst: 10.2.0.4             │                              │
    ├──── peering ──────────────►│                              │
    │                      NVA (10.0.0.4)                       │
    │                      IP forwarding ON                     │
    │                      패킷 검사 후 forward                 │
    │                            │  src: 10.1.0.4  ← 출발지 그대로 │
    │                            │  dst: 10.2.0.4               │
    │                            ├──── peering ────────────────►│
    │                            │                              │
    │                            │  Spoke-2 입장:               │
    │                            │  "Hub과 피어링인데,           │
    │                            │   Hub 주소(10.0.x.x)가 아닌  │
    │                            │   10.1.0.4에서 왔다?"         │
    │                            │  → forwarded traffic!        │

    Hub ↔ Spoke-2 Peering에서 allow_forwarded_traffic = false이면, Spoke-2는 이 패킷을 차단합니다. 출발지 IP가 Hub 주소가 아니라 "Hub이 전달(forward)한 제3자 트래픽"이기 때문입니다.

    allow_forwarded_traffic = true만으로는 부족합니다. Spoke 간 통신이 되려면 3가지가 모두 필요합니다:

    조건 역할 없으면
    UDR (Spoke 서브넷) 10.2.0.0/16 → NVA IP 경로 명시 Azure 기본 라우팅이 경로를 모름 → 통신 불가
    allow_forwarded_traffic = true (양쪽 peering) NVA가 forward한 패킷이 peering 통과 허용 NVA 거쳐 온 패킷이 peering에서 차단
    NVA IP forwarding NVA가 자신이 목적지가 아닌 패킷을 전달 NVA가 패킷 drop

    이 Lab에서 false인 이유

    이 Lab은 Hub ↔ Spoke 단 2개 VNet이고 NVA가 없습니다. Hub VM이 Spoke VM과 직접 통신하면 패킷 출발지는 Hub 주소(10.10.x.x) 그 자체이므로 "제3자가 forward한 트래픽"이 아닙니다. 따라서 false로 충분합니다.

    [참고] 3) allow_gateway_transit & 4) use_remote_gateways — Gateway 쌍 옵션¶

    이 두 옵션은 항상 쌍으로 사용됩니다. Hub에 VPN/ExpressRoute Gateway가 있을 때, Spoke가 해당 Gateway를 공유하기 위한 설정입니다.

    온프레미스
    │
    │ VPN / ExpressRoute
    ▼
Hub VNet
    VPN Gateway  ← allow_gateway_transit = true (Hub 측 peering)
    │
    │ peering
    ▼
Spoke VNet       ← use_remote_gateways = true  (Spoke 측 peering)
    Spoke VM이 Hub의 Gateway를 통해 온프레미스에 도달
    Hub 측 peering Spoke 측 peering 결과
    allow_gateway_transit = true use_remote_gateways = true Spoke가 Hub Gateway 공유
    false true 에러 (Hub이 허용 안 했는데 쓰려 함)
    true false Hub이 허용은 했지만 Spoke가 안 씀

    이 Lab에서 둘 다 false인 이유

    이 Lab에는 VPN/ExpressRoute Gateway가 없으므로 해당 없습니다.

    엔터프라이즈 환경에서의 VNet Peering

    Azure Virtual WAN (Secured Virtual Hub) 을 사용하는 엔터프라이즈 환경에서는, VNet Peering 자체를 중앙 랜딩존 팀이 Virtual Hub Connection으로 일괄 관리하며 Workload 팀이 직접 피어링을 만들지 않습니다. 위 4가지 옵션은 플랫폼이 자동 처리하므로 개발자가 직접 제어할 일이 없습니다.

    반면 팀이 직접 Hub-Spoke를 구성하는 환경(스타트업, PoC 등)에서는 위 옵션을 정확히 이해하고 설정해야 합니다.

    Private DNS Zone — VNet Link & Auto Registration¶

    Private DNS Zone의 VNet Link에는 registration_enabled 옵션이 있습니다.

    모드 registration_enabled 동작
    Resolution only (이 Lab) false A 레코드를 수동으로 등록 (azurerm_private_dns_a_record)
    Auto registration true VNet 안에 VM이 생성되면 자동으로 A 레코드 등록

    이 Lab에서는 registration_enabled = false로 설정하고 A 레코드를 Terraform으로 명시적으로 생성합니다. 이렇게 하면:

    • 레코드 이름을 직접 제어 (예: vm-hub, vm-spoke)
    • terraform plan 에서 DNS 변경을 코드로 추적 가능
    • Auto registration은 VM hostname(OS 내부 이름)으로 등록되어 이름이 예측 불가능할 수 있음

    Step-by-Step¶

    Terraform 코드 위치: labs/terraform/lab04-virtual-networking/

    1) Prepare variables¶

    예제 파일을 복사하고 본인 값을 입력합니다.

    cd labs/terraform/lab04-virtual-networking
cp terraform.tfvars.example terraform.tfvars

    에디터로 terraform.tfvars를 열고 다음을 설정합니다.

    • resource_group_name (필수) — 강사가 알려준 본인 RG 이름. 다른 사람 RG 사용 금지.
    • subscription_id (선택) — 비워 두면 현재 az context의 subscription이 사용됩니다.

    2) Login to Azure¶

    az login

    올바른 subscription이 선택되었는지 확인합니다.

    az account show -o table

    3) Initialize Terraform¶

    필요한 provider 플러그인(azurerm, random)을 다운로드합니다.

    terraform init
    예상 출력 
    Initializing the backend...
Initializing provider plugins...
- Finding hashicorp/azurerm versions matching "~> 4.0"...
- Finding hashicorp/random versions matching "~> 3.0"...
- Installing hashicorp/azurerm v4.x.x...
- Installing hashicorp/random v3.x.x...

Terraform has been successfully initialized!

    4) Plan¶

    배포할 리소스를 미리 확인합니다. VNet, subnet, NSG rule, VM 등이 기대한 대로 표시되는지 봅니다.

    terraform plan

    terraform plan에서 볼 것

    • 추가될 리소스 수 — VNet, subnet, NSG rule, VM 등이 기대와 맞는지
    • random_password — VM 비밀번호가 새로 생성되는지 (재적용 시 변경될 수 있음)

    5) Apply¶

    실제 Azure 리소스를 배포합니다.

    terraform apply

    프롬프트에 yes를 입력합니다. 완료까지 수 분~십여 분 소요될 수 있습니다(Bastion, VM 프로비저닝).

    예상 출력 (요약) 
    Apply complete! Resources: 36 added, 0 changed, 0 destroyed.

Outputs:
admin_password      = <sensitive>
admin_username      = "azureuser"
bastion_name        = "bas-az104-lab04"
hub_vm_private_ip   = "10.10.0.x"
location            = "koreacentral"
private_dns_zone    = "lab04.internal"
resource_group_name = "rg-az104-student01"
spoke_vm_private_ip = "10.20.0.x"
vnet_hub_name       = "vnet-az104-lab04-hub"
vnet_spoke_name     = "vnet-az104-lab04-spoke"

    총 36개 리소스: VNet x2, Subnet x3, Peering x2, NSG x2, NSG Rule x9 (hub: Allow SSH/ICMP/HTTP + Deny-all / spoke: Allow SSH from hub + Deny SSH from Bastion + Allow ICMP/HTTP + Deny-all), ASG x2, ASG-NIC Association x2, NIC x2, VM x2, Bastion x1, Public IP x1, Private DNS zone x1, DNS VNet link x2, DNS A record x2, subnet-NSG association x2, random_password x1

    6) Check outputs¶

    배포 결과를 확인합니다.

    terraform output
terraform output admin_username
terraform output -raw admin_password

    보안

    비밀번호는 터미널 기록이나 스크린샷에 남기지 마세요.

    Verify - Azure Portal¶

    1. Resource Group 이동 → 배포된 RG → 리소스 목록에 VNet x2, VM x2, Bastion, Private DNS zone 등이 보이는지 확인합니다.
    2. Bastion 리소스(bas-{prefix}) → Connect → SSH → 대상 VM을 선택합니다. 사용자명/비밀번호는 terraform output 값을 사용합니다.

    Basic SKU + VNet Peering

    Basic SKU는 피어링된 VNet의 VM에도 직접 연결할 수 있습니다. Hub의 Bastion 하나로 vm-hub와 vm-spoke 양쪽에 모두 접속 가능합니다.
    참고: Virtual network peering and Azure Bastion
    단, native client(로컬 SSH 클라이언트), shareable link, custom port 등 고급 기능은 Standard SKU 이상에서만 지원됩니다.

    NSG Deny 규칙으로 Bastion → Spoke 차단

    이 구성에서는 nsg-spoke에 AzureBastionSubnet(10.10.100.0/26)에서 오는 SSH를 차단하는 Deny 규칙(우선순위 101)을 추가하여, Bastion에서 vm-spoke로의 직접 연결을 의도적으로 차단합니다. Spoke VM에는 반드시 vm-hub에서 SSH Jump로만 접속해야 합니다. 이는 실무에서 Jump Server(점프 호스트) 패턴을 체험하기 위한 설계입니다.

    Bastion → vm-hub → vm-spoke (SSH Jump)¶

    NSG Deny 규칙

    Bastion에서 vm-spoke로의 직접 SSH 접속은 nsg-spoke의 Deny 규칙(우선순위 101)으로 차단됩니다. 반드시 vm-hub를 경유해야 합니다.

    1. Azure Portal에서 Bastion → Connect → 대상 VM: vm-hub → 사용자명/비밀번호 입력
    2. vm-hub 세션에서 Private DNS 이름으로 Spoke VM에 SSH Jump:
    # vm-hub 세션에서: Private DNS 이름으로 Spoke VM에 SSH
ssh <admin_username>@vm-spoke.lab04.internal
    • 호스트 키 확인 메시지가 나오면 yes 입력
    • 비밀번호는 Hub와 동일 (random_password 공유)

    vm-spoke.lab04.internal로 접속에 성공하면 VNet Peering과 Private DNS가 정상 동작하는 것입니다.

    Install nginx on vm-spoke¶

    vm-spoke 세션(SSH Jump 상태)에서 nginx를 설치하고 실행합니다.

    sudo apt update -y && sudo apt install -y nginx
sudo systemctl enable --now nginx

    설치 확인:

    curl -s http://localhost/ | head -5

    Welcome to nginx! 페이지가 출력되면 정상입니다. 이제 exit로 vm-hub 세션으로 돌아갑니다.

    exit

    Network Test (ICMP / HTTP 규칙 검증)¶

    vm-hub 세션에서 vm-spoke에 대한 네트워크 규칙 동작을 검증합니다.

    ICMP 허용 규칙 검증:

    ping -c 3 vm-spoke.lab04.internal
    예상 출력 
    PING vm-spoke.lab04.internal (10.20.1.x) 56(84) bytes of data.
64 bytes from 10.20.1.x: icmp_seq=1 ttl=64 time=1.xx ms
64 bytes from 10.20.1.x: icmp_seq=2 ttl=64 time=0.xx ms
64 bytes from 10.20.1.x: icmp_seq=3 ttl=64 time=0.xx ms

    HTTP 허용 규칙 검증 (ASG 대상 포트 80):

    curl -sS -o /dev/null -w "HTTP Status: %{http_code}\n" http://vm-spoke.lab04.internal/
    예상 출력 
    HTTP Status: 200

    nginx 페이지 전체를 확인하려면:

    curl -s http://vm-spoke.lab04.internal/ | head -5
    예상 출력 
    <!DOCTYPE html>
<html>
<head>
<title>Welcome to nginx!</title>

    규칙 동작 이해

    • ICMP: nsg-spoke 의 Allow ICMP 규칙(우선순위 110)이 hub 워크로드 서브넷에서 오는 ping을 허용합니다.
    • HTTP: nsg-spoke 의 Allow HTTP 규칙(우선순위 120)이 asg-app-spoke 멤버(vm-spoke)의 포트 80 트래픽을 허용합니다.
    • SSH: nsg-spoke 의 Allow SSH from hub(우선순위 100)가 hub 워크로드 서브넷 발 SSH만 허용하고, Deny SSH from Bastion(우선순위 101)이 Bastion 직접 접속을 차단합니다.

    추가 확인 (Portal)

    아래 항목을 Portal에서 직접 확인해 보세요:

    • VNet Peering: 각 VNet → Peerings 탭에서 Connected 상태인지 확인
    • Private DNS zone: 리소스 그룹에서 DNS zone → Record sets 에서 A 레코드 확인, Virtual network links 에서 hub/spoke 둘 다 연결되어 있는지 확인
    • NSG: 각 NSG → Inbound/Outbound security rules 에서 규칙 목록과 우선순위 확인
    • ASG: 각 NIC → IP configurations → Application security groups 에서 ASG 연결 확인

    Cleanup (Teardown)¶

    terraform destroy

    프롬프트에 yes를 입력합니다.

    비용 방지

    Bastion과 VM은 시간당 과금됩니다. 실습이 끝나면 반드시 terraform destroy를 실행하세요.

    Terraform Files¶

    파일 역할
    versions.tf Terraform / provider 버전 제약
    providers.tf azurerm provider, random provider
    variables.tf prefix, RG, 주소 공간, VM 크기 등 입력 변수
    main.tf RG data source, random_password, locals
    network.tf VNet, Subnet, Peering
    security.tf NSG, NSG rules, ASG, subnet-NSG 연결
    dns.tf Private DNS zone, VNet link, A record
    bastion.tf Bastion Host + Public IP
    vm.tf Hub VM, Spoke VM (Ubuntu 24.04)
    outputs.tf VNet/VM/DNS 출력
    terraform.tfvars.example 변수 예시 파일 (비밀 값 없음)

    Troubleshooting¶

    증상 원인 해결
    Resource Group "xxx" was not found RG 이름 오류 또는 다른 subscription terraform.tfvars와 az account show 확인
    Bastion 배포 실패 AzureBastionSubnet이 /26 미만 기본 값 그대로 사용 권장 (10.10.100.0/26)
    VM SSH 접속 불가 NSG 규칙 누락 또는 Bastion 미배포 az network nsg rule list로 규칙 확인
    terraform destroy 시간 초과 Bastion/VM 삭제 지연 재실행하거나 Portal에서 수동 삭제 후 terraform state rm
