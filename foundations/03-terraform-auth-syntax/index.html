    Terraform 인증, 문법 입문 (Azure)¶

    블록별 코드 형태는 Terraform HCL — 블록 샘플을 먼저 봐도 됩니다.

    이 문서는 개념, 문법 틀만 잡습니다. 할 일 순서, 명령 예시는 Azure 접근 및 Terraform 인증과 Terraform Lab을 따르세요.

    이 과정에서의 인증¶

    인증 흐름¶

    단계 내용
    사용자 Microsoft Entra 사용자로 Azure CLI에 로그인 (az login)
    Terraform AzureRM provider가 같은 CLI 세션의 자격 증명을 사용해 Azure API 호출
    권한 범위 강사가 부여한 할당 Resource Group 안에서 Contributor 수준으로 동작한다고 가정

    인증 방식 비교 (참고)¶

    방식 사용처 이 과정
    Azure CLI (az login) 개발자 로컬, 교육 ✓ 사용
    Service Principal (Client Secret) CI/CD 파이프라인 참고만
    Managed Identity Azure VM, AKS 위에서 실행 참고만
    OIDC (Federated Credential) GitHub Actions 등 참고만

    다른 인증 방식

    Service Principal, Managed Identity, OIDC 등은 운영, CI에서 흔하지만, 본 과정 실습에서는 다루지 않거나 참고만 합니다.

    최소 문법 (읽을 수 있으면 됨)¶

    Block 구조¶

    resource "azurerm_resource_group" "example" {
  name     = var.resource_group_name
  location = var.location
}
    • 첫 줄: resource type과 label(Terraform 내부 이름).
    • 본문: argument name = value 형태 (문자열은 따옴표, 참조는 var.xxx 등).

    자주 보는 참조¶

    형태 의미 예시
    var.<name> variables로 받은 값 var.resource_group_name
    local.<name> locals 블록의 계산값 local.tags
    <resource_type>.<label>.<attr> 다른 리소스의 attribute azurerm_virtual_network.hub.id
    data.<type>.<label>.<attr> data source의 attribute data.azurerm_resource_group.this.location

    data 블록 (조회만)¶

    data "azurerm_resource_group" "existing" {
  name = var.resource_group_name
}

    이 과정 skeleton은 이미 있는 RG를 data로 읽고 연결만 검증하는 패턴을 사용합니다.¶

    terraform.tfvars — 변수 값 파일¶

    실습에서는 terraform.tfvars.example을 복사하여 사용합니다.

    cp terraform.tfvars.example terraform.tfvars

    # terraform.tfvars
resource_group_name = "rg-az104-student01"
# subscription_id   = ""   # 비우면 az CLI 현재 구독

    .gitignore에 포함

    terraform.tfvars에는 개인 정보(RG 이름 등)가 들어갈 수 있습니다. .gitignore에 *.tfvars를 추가하여 Git에 올리지 않도록 합니다. terraform.tfvars.example은 비밀 값 없이 템플릿으로 commit합니다.

    의존성 (Dependency)¶

    Terraform은 리소스 간 참조 관계를 자동으로 파악하여 생성/삭제 순서를 결정합니다.

    암묵적 의존성 (Implicit)¶

    attribute를 참조하면 자동으로 순서가 정해집니다.

    resource "azurerm_virtual_network" "hub" {
  name                = "vnet-hub"
  address_space       = ["10.10.0.0/16"]
  resource_group_name = data.azurerm_resource_group.this.name
  location            = data.azurerm_resource_group.this.location
}

resource "azurerm_subnet" "workload" {
  name                 = "snet-workload"
  virtual_network_name = azurerm_virtual_network.hub.name   # ← hub VNet 참조
  resource_group_name  = data.azurerm_resource_group.this.name
  address_prefixes     = ["10.10.1.0/24"]
}
# Terraform은 VNet → Subnet 순서로 생성, 삭제 시 역순

    명시적 의존성 (depends_on)¶

    attribute 참조 없이도 순서가 필요한 경우 (예: Bastion 배포 완료 후 Peering):

    resource "azurerm_virtual_network_peering" "hub_to_spoke" {
  # ... peering 설정 ...

  depends_on = [azurerm_bastion_host.this]  # Bastion 완료 대기
}

    대부분 암묵적으로 충분

    depends_on은 attribute 참조가 없는 특수한 경우에만 사용합니다. Lab 04의 Peering ↔ Bastion 관계가 대표적인 예입니다.

    기타 문법 맛보기¶

    이 과정 Lab에서 직접 쓰지는 않지만, 코드를 읽다 보면 만날 수 있는 패턴입니다.

    문법 용도 예시
    count 같은 리소스를 N개 생성 count = 3 → resource[0], resource[1], resource[2]
    for_each map/set 기반으로 리소스 생성 for_each = toset(["hub", "spoke"])
    dynamic 블록 내부를 동적으로 반복 NSG rule 여러 개를 변수에서 생성
    삼항 연산자 조건부 값 var.location != "" ? var.location : data...location
