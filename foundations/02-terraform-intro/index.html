    Terraform 개요¶

    Terraform은 HashiCorp의 infrastructure provisioning 도구로, HCL(HashiCorp Configuration Language)로 원하는 인프라를 선언하고, plan → apply로 실제 환경에 반영합니다.

    핵심 개념¶

    용어 설명
    Provider 어떤 platform API와 말할지 정의하는 plugin. Azure에서는 azurerm 등을 사용합니다.
    Resource 만들거나 관리할 대상(예: Resource Group, Virtual Network). resource 블록으로 기술합니다.
    Data source 만들지 않고 이미 있는 리소스를 조회할 때 쓰는 블록 (data). 이 과정 skeleton에서는 할당된 RG를 data로 읽습니다.
    State Terraform이 현재 어떤 리소스를 관리 중인지 저장한 기록. 기본은 로컬 terraform.tfstate 파일.
    Module 재사용을 위해 묶어 둔 .tf 묶음. 처음에는 root module만 써도 충분합니다.

    Lifecycle — 전체 흐름¶

    terraform init¶

    terraform init
    • .terraform/ 디렉터리에 provider plugin 다운로드
    • .terraform.lock.hcl — provider 버전 잠금 (팀 협업 시 commit 권장)
    • backend 설정이 있으면 remote state 연결

    terraform plan¶

    terraform plan
    • 현재 state와 .tf 코드를 비교하여 변경 예상을 보여줌
    • 실제 환경에는 아무 변경 없음 — 안전하게 검토 가능
    plan 출력 읽는 법 
    Terraform will perform the following actions:

  # azurerm_virtual_network.hub will be created
  + resource "azurerm_virtual_network" "hub" {
      + address_space       = ["10.10.0.0/16"]
      + id                  = (known after apply)
      + name                = "vnet-az104-lab04-hub"
      ...
    }

Plan: 36 to add, 0 to change, 0 to destroy.
    기호 의미
    + 새로 생성
    ~ 기존 리소스 변경 (in-place update)
    - 삭제
    -/+ 삭제 후 재생성 (replacement — 주의!)

    terraform apply¶

    terraform apply
    • plan 결과를 보여주고 yes 확인 후 실제 Azure API 호출
    • 완료 후 terraform.tfstate 업데이트

    terraform destroy¶

    terraform destroy
    • state에 기록된 모든 리소스를 역순으로 삭제
    • 실습 후 비용 방지를 위해 반드시 실행

    State — 왜 중요한가?¶

    State 파일(terraform.tfstate)은 Terraform이 "내가 지금 뭘 관리하고 있는지" 기억하는 핵심 파일입니다.

    코드(.tf)     ←→     State 파일     ←→     실제 Azure 환경
  원하는 상태          현재 기록            실제 리소스
    역할 설명
    매핑 .tf 코드의 resource "azurerm_virtual_network" "hub" ↔ Azure의 실제 VNet ID
    Diff 계산 plan 시 코드 vs state를 비교하여 생성/변경/삭제 판단
    성능 매번 Azure API를 전부 조회하지 않고 state 캐시 활용
    의존성 리소스 간 참조 관계를 기록하여 삭제 순서 결정

    State 파일 주의사항

    • terraform.tfstate에는 비밀번호, 키 등이 평문으로 포함될 수 있음
    • .gitignore에 반드시 추가 — Git에 올리면 안 됨
    • 팀 작업 시 remote backend(Azure Storage, S3 등)로 공유 + locking

    교육용 로컬 state vs 운영 remote state¶

    항목 로컬 state (이 과정) Remote state (운영)
    위치 작업 디렉터리 terraform.tfstate Azure Storage, S3, Terraform Cloud 등
    공유 본인 PC에서만 팀 전체가 같은 state 참조
    동시 실행 보호 없음 (1인 작업 전제) Locking — 동시 apply 방지
    백업 수동 자동 버전 관리

    .terraform/ 디렉터리 구조¶

    terraform init 후 생성되는 구조입니다.

    lab04-virtual-networking/
├── .terraform/
│   ├── providers/
│   │   └── registry.terraform.io/
│   │       └── hashicorp/
│   │           ├── azurerm/    ← provider binary
│   │           └── random/
│   └── terraform.tfstate       ← backend 메타데이터
├── .terraform.lock.hcl         ← provider 버전 잠금
├── terraform.tfstate           ← state (apply 후 생성)
├── terraform.tfstate.backup    ← 이전 state 백업
├── terraform.tfvars            ← 본인 변수값 (gitignore)
├── variables.tf
├── main.tf
└── ...

    .terraform.lock.hcl

    이 파일은 provider 버전과 해시를 고정합니다. 팀에서는 Git에 commit하여 모두 같은 provider 버전을 사용하도록 합니다.

    Terraform vs ARM/Bicep¶

    비교 항목 Terraform ARM Template Bicep
    언어 HCL JSON Bicep DSL
    상태 관리 State 파일 (명시적) Azure Deployment 기록 Azure Deployment 기록
    멀티 클라우드 ✓ (AWS, GCP, Azure 등) ✗ Azure only ✗ Azure only
    Dry-run terraform plan what-if what-if
    에코시스템 Registry, 커뮤니티 모듈 제한적 제한적
    학습 곡선 HCL 문법 학습 필요 JSON 구조 복잡 Azure 사용자에게 친숙

    입력, 출력 (variables / output)¶

    • Input: variables.tf, terraform.tfvars 등으로 환경마다 바꿀 값(RG 이름, 지역 등).
    • Output: output으로 배포 후 확인할 값(리소스 ID 등)을 출력할 수 있습니다.

    블록별 짧은 HCL 예시는 Terraform HCL — 블록 샘플을 참고하세요.
    인증, 실습 순서는 인증, 문법 입문에서 다룹니다.
