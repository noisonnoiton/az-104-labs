    IaC 패러다임 개요¶

    인프라를 다루는 방식은 여러 패러다임으로 나눌 수 있습니다. 이름은 자료마다 조금씩 다르지만, 아래 네 가지로 이해하면 Azure, Terraform 강의 흐름을 잡기 좋습니다.

    네 가지 접근¶

    패러다임 한 줄 설명 예시
    Ad-Hoc 필요할 때마다 사람이 콘솔, 원격으로 직접 조작 포털에서 클릭, SSH로 수동 설정
    구성 관리 (Configuration Management) 이미 있는 서버 내부 상태(패키지, 파일, 서비스)를 원하는 상태로 맞춤 Ansible, Chef, Puppet
    서버 템플릿 (Server templating) 배포 이전에 OS, middleware가 박힌 이미지/템플릿을 만들어 두고, 동일한 VM을 반복 생성 Packer, 커스텀 VM 이미지
    Infrastructure provisioning 네트워크, VM, 스토리지 등 클라우드 리소스 자체를 API, 코드로 생성, 변경, 삭제 ARM/Bicep, Terraform, 네이티브 SDK

    진화 흐름¶

    • Ad-Hoc → 구성 관리: 수동 반복을 줄이기 위해 서버 내부 상태를 코드로 관리
    • 구성 관리 → 서버 템플릿: 설정 드리프트를 원천 차단 — 매번 같은 이미지로 새로 만듦
    • 서버 템플릿 → Infrastructure provisioning: 서버뿐 아니라 네트워크, 스토리지, 권한까지 코드로

    이 과정에서의 위치

    Terraform Lab은 주로 infrastructure provisioning(리소스를 선언적으로 만들고 버전에 맞게 적용)에 해당합니다. 앱 설정만 맞추는 구성 관리와 역할이 다르다는 점만 구분해 두면 됩니다.

    선언형 vs 명령형¶

    IaC 도구는 크게 선언형(Declarative)과 명령형(Imperative) 두 스타일로 나뉩니다.

    구분 선언형 명령형
    방식 원하는 상태를 기술 → 도구가 현재와 비교하여 차이만 반영 실행 순서를 하나씩 지시 → 위에서 아래로 수행
    예시 Terraform, ARM/Bicep, Kubernetes YAML Azure CLI 스크립트, PowerShell, Bash
    장점 멱등성 보장, 상태 추적 용이, diff 가능 직관적, 빠른 프로토타입
    단점 학습 곡선, 상태 파일 관리 필요 실행 순서 의존, 재실행 시 중복/오류 가능 
    # 명령형: "이 순서대로 해라"
az network vnet create --name myVNet --address-prefixes 10.0.0.0/16 ...
az network vnet subnet create --name mySubnet ...
az network nsg create --name myNSG ...

# 선언형: "이 상태가 되어라"
resource "azurerm_virtual_network" "this" {
  name          = "myVNet"
  address_space = ["10.0.0.0/16"]
}

    멱등성 (Idempotency)

    같은 코드를 여러 번 apply해도 결과가 동일합니다. 이미 원하는 상태이면 Terraform은 No changes를 반환합니다. 이것이 선언형의 핵심 장점이며, 수동 작업에서 발생하는 설정 드리프트(drift)를 방지합니다.

    Azure 도구 비교 — 어디에 해당하나?¶

    도구 유형 상태 관리 멀티 클라우드 이 과정에서
    Azure Portal Ad-Hoc (GUI) 없음 ✗ 검증, 확인용
    Azure CLI / PowerShell 명령형 스크립트 없음 ✗ 검증, 빠른 조회
    ARM Template 선언형 (JSON) Azure 배포 기록 ✗ 참고만
    Bicep 선언형 (Azure DSL) Azure 배포 기록 ✗ 참고만
    Terraform 선언형 (HCL) State 파일 ✓ 실습 주력

    왜 Terraform?

    이 과정에서 Terraform을 선택한 이유:

    1. plan → apply 워크플로가 명확하여 변경 전 검토가 자연스러움
    2. State가 코드와 실제 환경의 차이를 추적해 줌
    3. Azure 외 다른 클라우드에서도 동일한 경험 재사용 가능
    4. 에코시스템(Registry, 커뮤니티 모듈)이 풍부

    Azure와의 연결 (강의 맥락)¶

    • ARM / Bicep — Azure "네이티브" 선언형 템플릿. Azure 배포 기록(Deployment)이 자동으로 남음.
    • Terraform — Azure를 포함 여러 클라우드에 같은 workflow(plan → apply)로 쓰기 쉽게 provider 모델을 둔 도구. State 파일을 별도 관리해야 하는 대신 멀티 클라우드, 세밀한 diff가 장점.

    자세한 Terraform 용어는 Terraform 개요로 이어집니다.
