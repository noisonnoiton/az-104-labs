    Kubernetes 핵심 개념 (참고)¶

    이 문서의 역할

    2일차 AKS Lab(Lab 90~Lab 92) 전에 Kubernetes 핵심 용어와 구조를 한 페이지로 훑습니다. 실습에서 직접 만나는 개념 위주로 정리했으며, 깊이 있는 내용은 공식 문서를 참고하세요.

    1. 인프라 진화 — 서버에서 오케스트레이션까지¶

    단계 특징 비유
    Physical Server 1 앱 = 1 장비 단독 주택
    Virtual Machine 1 장비에 여러 OS 아파트
    Container 경량, 빠른 시작 공유 오피스
    Kubernetes 컨테이너 자동 관리 건물 관리자

    컨테이너는 패키징을 해결하고, Kubernetes는 대규모 운영을 해결합니다.

    2. Kubernetes란?¶

    Kubernetes(K8s)는 컨테이너화된 애플리케이션의 배포, 스케일링, 자동 복구를 관리하는 오픈소스 플랫폼입니다.

    항목 내용
    유형 Open Source (CNCF)
    카테고리 Container Orchestrator
    기원 Google Borg → 2014 공개
    약칭 K8s (K와 s 사이 8글자)

    핵심 원칙 — Declarative(선언적):
    "원하는 상태(desired state)"를 YAML로 정의하면, Kubernetes가 현재 상태를 원하는 상태로 수렴시킵니다.

    3. Architecture¶

    Control Plane (마스터)¶

    Component 역할
    API Server 모든 요청의 진입점 (kubectl 명령 수신)
    etcd 클러스터 상태를 저장하는 Key-Value DB
    Scheduler 새 Pod을 어느 노드에 배치할지 결정
    Controller Manager desired state ↔ current state를 지속 비교·조정

    AKS에서는

    Control Plane은 Azure가 관리합니다. etcd에 직접 접근하거나 마스터 노드에 SSH 할 수 없으며, kubectl로만 상호 작용합니다.

    Worker Node¶

    Component 역할
    kubelet API Server 지시를 받아 Pod/컨테이너 실행
    kube-proxy Service 네트워킹 규칙 관리 (iptables/IPVS)
    Container Runtime 실제 컨테이너 실행 (AKS는 containerd 사용)

    AKS Node Pool

    Lab 90에서 system pool(1 node) + user pool(1 node)을 생성합니다. VMSS 기반이며 kubectl get nodes로 확인합니다.

    4. Core Objects¶

    Pod¶

    • Kubernetes의 최소 배포 단위
    • 보통 1 Pod = 1 Container (가장 흔한 패턴)
    • Pod 안 컨테이너끼리 네트워크(같은 IP, localhost 통신)와 스토리지를 공유
    • Pod은 임시(ephemeral) — 죽으면 새로 생성 (IP 변경)

    Namespace¶

    클러스터를 논리적으로 격리하는 단위입니다.

    kubectl get namespaces
    Namespace 용도
    default 기본
    kube-system K8s 핵심 컴포넌트 (CoreDNS, Metrics Server 등)
    app-routing-system AKS Web App Routing (Ingress Controller)

    Lab 91에서

    마이크로서비스를 awesome, awesome-infra 등 별도 namespace에 배포합니다.

    Label & Selector¶

    • Label: 리소스에 붙이는 key=value 태그 (예: app=nginx)
    • Selector: Label을 기준으로 리소스를 쿼리 (Service가 Pod을 찾는 방식)

    5. Workloads¶

    Type 용도 Lab 연결
    Deployment Stateless 앱 (Web, API) — ReplicaSet 자동 관리 Lab 91 마이크로서비스 전체
    StatefulSet 순서·고유 ID 필요 (DB, 메시지 큐) Lab 91 MySQL, RabbitMQ
    DaemonSet 모든 노드에 1개씩 (모니터링 agent, 로그 수집) Lab 92 OTel Collector
    Job / CronJob 1회·주기적 배치 작업 (이 과정에서는 직접 사용하지 않음)

    Deployment → ReplicaSet → Pod

    Deployment를 만들면 K8s가 자동으로 ReplicaSet을 생성하고, ReplicaSet이 지정된 수만큼 Pod을 유지합니다. replicas: 3이면 항상 3개 Pod이 Running 상태를 유지합니다.

    6. Service & Ingress¶

    Service Type 노출 범위 사용 예
    ClusterIP 클러스터 내부만 서비스 간 통신 (DB → App)
    NodePort 노드 IP:포트 테스트용 (프로덕션 비권장)
    LoadBalancer 외부 IP (Azure LB 자동 생성) Lab 91 API Gateway 외부 노출
    ExternalName 외부 DNS 매핑 외부 SaaS 연결

    Ingress¶

    • L7(HTTP/HTTPS) 라우팅 — Host, Path 기반으로 여러 Service에 트래픽 분배
    • AKS에서는 Web App Routing add-on이 NGINX Ingress Controller를 자동 배포
    • IngressClass: webapprouting.kubernetes.azure.com

    Lab 91에서

    Ingress를 통해 <EXTERNAL-IP>.nip.io host-based routing으로 서비스를 외부 노출합니다.

    DNS¶

    클러스터 내 Service는 자동으로 DNS 이름을 받습니다:

    <service-name>.<namespace>.svc.cluster.local

    7. Storage & Configuration¶

    Storage¶

    개념 역할
    Volume Pod에 마운트되는 저장소 (Pod 삭제 시 소멸 가능)
    PersistentVolume (PV) 클러스터 수준의 영구 스토리지
    PersistentVolumeClaim (PVC) Pod이 PV를 요청하는 방법
    StorageClass 동적 프로비저닝 규칙 (AKS: managed-csi = Azure Disk)

    Lab 91에서

    MySQL, RabbitMQ가 PVC를 사용하여 Azure Managed Disk에 데이터를 영구 저장합니다.

    ConfigMap & Secret¶

    Object 용도
    ConfigMap 비민감 설정 (DB host, feature flag)
    Secret 민감 데이터 (password, API key) — base64 인코딩

    Health Probes¶

    Probe 역할
    Liveness 죽었으면 재시작
    Readiness 준비 안 됐으면 트래픽 차단
    Startup 초기화 완료까지 다른 probe 지연

    8. Scaling¶

    • 트래픽이 늘면 같은 Pod에 컨테이너를 추가하는 것이 아니라, Pod 수를 늘립니다 (Horizontal Scaling)
    • 노드가 부족하면 새 노드를 추가합니다 (Cluster Autoscaler)
    방식 대상 설명
    HPA (Horizontal Pod Autoscaler) Pod CPU/Memory 기준 Pod 수 자동 조정
    VPA (Vertical Pod Autoscaler) Pod 개별 Pod의 resource request/limit 조정
    Cluster Autoscaler Node 노드 수 자동 조정 (AKS에서 지원)

    9. Monitoring (Lab 92 연결)¶

    도구 역할 Lab 연결
    kubectl top nodes/pods 실시간 리소스 사용량 Lab 90 Deep Dive
    kubectl get events 클러스터 이벤트 조회 트러블슈팅 전반
    OTel Collector Traces, Metrics, Logs 수집·전송 Lab 92
    Azure Monitor Application Insights, Log Analytics Lab 92

    Lab 92에서

    OTel Collector를 DaemonSet으로 배포하고, Azure Monitor exporter로 분산 추적(Distributed Tracing)을 확인합니다.

    10. kubectl Cheatsheet (Lab에서 쓰는 명령)¶

    명령 설명
    kubectl get nodes 노드 목록
    kubectl get pods -A 전체 namespace Pod 조회
    kubectl get svc Service 목록 (External IP 확인)
    kubectl get namespaces Namespace 목록
    kubectl describe pod <NAME> Pod 상세 (Events, Conditions)
    kubectl logs <POD> Pod 로그 확인
    kubectl top nodes 노드 CPU/Memory 사용량
    kubectl top pods -A Pod CPU/Memory 사용량
    kubectl get events --sort-by='.lastTimestamp' 최신 이벤트
    kubectl apply -f <FILE> 선언적 리소스 생성/업데이트
    kubectl apply -k <DIR> Kustomize 기반 배포
    kubectl delete -f <FILE> 리소스 삭제
    kubectl exec -it <POD> -- bash Pod 내부 셸 접속
    kubectl rollout undo deploy <NAME> 이전 버전 롤백
    kubectl scale deploy <NAME> --replicas=N 수동 스케일링
